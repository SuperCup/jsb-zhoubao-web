# 发布流程

本项目采用“测试先行、正式发布需群内确认”的发布规则。

## 发布地址

- 测试地址：`https://supercup.github.io/jsb-zhoubao-web/`
- 正式地址：`https://agent.ismartgo.com/carlsberg/weekly`
- Netlify 已弃用，不再作为本项目发布通道。

## 默认发布规则

- 所有代码、数据和看板展示修改，默认只发布到测试地址。
- 推送 `main` 后，GitHub Actions workflow `Deploy Test GitHub Pages` 会自动构建静态包，并更新 `gh-pages` 分支。
- `gh-pages` 分支只作为测试发布源使用，不代表正式地址已更新。
- 未经群成员验证通过并明确允许，不得更新正式地址。

## 正式发布门禁

正式地址发布必须同时满足：

1. 测试地址已完成发布并可访问。
2. 群成员已在群聊中完成验证。
3. 群成员明确回复允许发布到正式地址。

满足以上条件后，才允许通过 `agent.ismartgo.com` 上传接口发布到正式地址。

## 操作清单

1. 修改代码或数据。
2. 本地验证：至少运行 `npm run lint`、`npm run build`、`npm run build:github`。
3. 推送 `main`，等待测试地址更新。
4. 在群聊中发送测试地址和变更说明，等待群成员验证。
5. 收到明确允许后，再发布正式地址。
6. 发布正式地址后，在群聊中回报正式地址、发布时间和对应提交。

## 正式发布方式

正式环境使用 `agent.ismartgo.com` 的网页包上传接口，不再更新 Netlify。

构建正式包时必须使用相对路径作为 Vite base，确保 ZIP 可被 `agent.ismartgo.com` 解压到对应网页包目录。入口页内置无斜杠兼容逻辑，访问 `https://agent.ismartgo.com/carlsberg/weekly` 时会先跳转到 `https://agent.ismartgo.com/carlsberg/weekly/`，再加载相对资源。

```bash
PUBLIC_BASE_PATH=./ npm run build:public
rm -f /tmp/carlsberg-weekly-formal.zip
(cd public-dist && zip -qr /tmp/carlsberg-weekly-formal.zip .)
```

上传接口：

```text
https://agent.ismartgo.com/carlsberg/weekly/upload?token=<当天上传 token>
```

上传表单：

- `file`: ZIP 文件，字段名必须为 `file`
- `title`: 可选，建议填 `嘉士伯即时零售经营周报`

上传后至少验证：

```text
https://agent.ismartgo.com/carlsberg/weekly
https://agent.ismartgo.com/carlsberg/weekly/
https://agent.ismartgo.com/carlsberg/weekly/data/dashboard-data.json
```

当天上传 token 会变化，不得写入仓库。

## 禁止事项

- 不得把正式发布作为默认动作。
- 不得在群成员验证前发布正式地址。
- 不得把 `agent.ismartgo.com` 的账号、密码、token 或其他密钥写入仓库。
